## scope

the extension surface: a typed hook bus, trait-based tool/command registries, the `Extension` trait, the `ExtensionApi`, and two extension languages: rust crates (compiled in, first-party) and rhai scripts (runtime-loaded, user-authored). this is the spine everything else hangs off.

### in scope

1. `HookBus` — typed pub/sub for lifecycle events.
2. `Tool` trait + `ToolRegistry` — dynamic registration, active-tool sets, schema export.
3. `Command` trait + `CommandRegistry` — slash commands.
4. `Extension` trait — rust crates compiled in that register tools, commands, hooks at startup.
5. `ScriptExtension` — rhai scripts loaded at runtime that register tools, commands, hooks at startup.
6. `ExtensionApi` — the one registration surface shared by rust and rhai.

### not in scope

- specific first-party extensions (v2.1–v2.9).
- the rhai api surface details — see `v2.10-scripting.md`.

## design rationale (mined from pi)

pi's `AgentHarness` (`packages/agent/src/harness/agent-harness.ts`) is the model. it exposes a single `hooks.emit(event)` call from the loop and lets external code subscribe via `on(type, handler)` or `observe(handler)`. the harness itself stores no extension policy.

the key split pi gets right:
- `observe()` — read-only, sees every event, return ignored.
- `on(type, handler)` — participates in that event's semantics, can mutate/transform/block.
- `emit(event)` — the *only* thing the loop calls.

pi's `ExtensionAPI` (`packages/coding-agent/src/core/extensions/types.ts`) exposes ~25 event types. each event carries its own result type via a phantom `HookResult` symbol. we mirror that with rust's type system: `HookEvent` trait with an associated `Result` type.

what we drop from pi: the ts runtime adapter concerns (`AsyncLocalStorage`, `diagnostics_channel`), the `ExtensionFactory` async init dance, the `defineTool` type gymnastics, the `ExtensionRuntime`/`ExtensionActions`/`ExtensionContextActions` split. rust doesn't need any of it — a trait + a `HashMap<EventType, Vec<Handler>>` does the job. plus a rhai runtime for scripts.

what we add that pi doesn't have: an embedded scripting tier. pi has js/ts only. inout has rust crates for deep integration and rhai scripts for user-facing extensibility.

## components

### hook bus

file: `crates/inout-core/src/hooks.rs`

```rust
pub trait HookEvent: Send + 'static {
    type Result: Default + Send;
    const TYPE: &'static str;
}

pub type HookHandler<E> = Box<dyn Fn(&E, &mut HookCtx) -> E::Result + Send + Sync>;

pub struct HookBus {
    observers: Vec<Box<dyn Fn(&dyn Any) + Send + Sync>>,
    handlers: HashMap<&'static str, Vec<Box<dyn Fn(&dyn Any, &mut HookCtx) -> Any + Send + Sync>>,
}

impl HookBus {
    pub fn observe<F>(&self, f: F) -> SubscriptionId
        where F: Fn(&dyn Any) + Send + Sync + 'static;
    pub fn on<E: HookEvent, F>(&mut self, f: F) -> SubscriptionId
        where E: HookEvent, F: Fn(&E, &mut HookCtx) -> E::Result + Send + Sync + 'static;
    pub async fn emit<E: HookEvent>(&self, event: E) -> E::Result;
    pub fn clear(&mut self);
}
```

`HookCtx` carries references to session-relevant state: `config`, `tools`, `commands`, `session`. handlers run in registration order. transform events chain (handler B sees A's output). block events early-exit on first `block: true`.

### events (initial set)

| event | result | when |
|---|---|---|
| `before_agent_start` | `{ messages, system_prompt }` | before loop, can inject messages / rewrite system prompt |
| `context` | `{ messages }` | before each llm call, can transform messages |
| `before_provider_request` | `{ stream_options_patch }` | per-request options (headers, timeout, transport) |
| `before_provider_payload` | `{ payload }` | the final wire payload, mutable — the raw-stream hook (v2.1) |
| `before_provider_headers` | `()` | mutate header map in place |
| `after_provider_response` | `()` | status + headers, before stream consumed |
| `tool_call` | `{ block, reason }` | before execution, can block |
| `tool_result` | `{ content, details, is_error, terminate }` | after execution, can patch |
| `message_end` | `()` | assistant message complete |
| `turn_end` | `()` | turn done |
| `agent_end` | `()` | loop settled |
| `session_start` | `()` | session loaded or created |
| `session_end` | `()` | session persisted/shutdown |
| `model_select` | `{ model }` | before llm call, can change model |

### tool trait

file: `crates/inout-core/src/tools.rs`

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &'static str;
    fn schema(&self) -> Value;
    async fn run(&self, args: Value) -> Result<String, ToolError>;

    fn permission_class(&self) -> PermissionClass { PermissionClass::Read }
    fn affected_path(&self, _args: &Value) -> Option<PathBuf> { None }
    fn retry_safe(&self) -> bool { false }
    fn shows_inline_output(&self) -> bool { true }
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum PermissionClass { Read, Write, Shell, Spawn, Network }
```

`PermissionClass` is defined in `inout-core` because both the `Tool` trait and the `permissions` extension need it. `affected_path` lets hooks (e.g. jail) inspect the target before execution.

### tool registry

file: `crates/inout-core/src/tools.rs`

```rust
pub struct ToolRegistry {
    tools: HashMap<String, Arc<dyn Tool>>,
    active: Vec<String>,
}

impl ToolRegistry {
    pub fn register<T: Tool>(&mut self, tool: T);
    pub fn unregister(&mut self, name: &str);
    pub fn set_active(&mut self, names: Vec<String>) -> Result<()>;
    pub fn active_schemas(&self) -> Vec<Value>;
    pub async fn dispatch(&self, name: &str, args: Value) -> Result<String, ToolError>;

```

`set_active` validates names exist and are unique — rejects duplicates, same as pi's `validateUniqueNames`. active set is per-session; changing it is a session action persisted by `inout-ext-sessions`.

### command registry

file: `crates/inout-core/src/commands.rs`

```rust
pub struct Command {
    pub name: &'static str,
    pub description: &'static str,
    pub handler: Box<dyn Fn(&mut CommandCtx, Vec<String>) -> BoxFuture<()> + Send + Sync>,
}

pub struct CommandCtx<'a> {
    pub session: &'a mut Session,
    pub hooks: &'a HookBus,
    pub tools: &'a mut ToolRegistry,
    pub config: &'a mut Config,
}

pub struct CommandRegistry { commands: HashMap<String, Command> }

impl CommandRegistry {
    pub fn register(&mut self, cmd: Command);
    pub async fn dispatch(&self, name: &str, args: Vec<String>, ctx: &mut CommandCtx);
    pub fn names(&self) -> Vec<&str>;
}
```

commands are async and have full session mutation access. slash-command parsing lives in the binary crate (or a future tui extension), not in core.

### extension api

file: `crates/inout-core/src/extension.rs`

```rust
pub struct ExtensionApi<'a> {
    pub tools: &'a mut ToolRegistry,
    pub commands: &'a mut CommandRegistry,
    pub hooks: &'a mut HookBus,
    pub config: &'a mut Config,
    pub session: &'a mut Session,
}

impl<'a> ExtensionApi<'a> {
    pub fn new(
        tools: &'a mut ToolRegistry,
        commands: &'a mut CommandRegistry,
        hooks: &'a mut HookBus,
        config: &'a mut Config,
        session: &'a mut Session,
    ) -> Self;
}
```

this is the single surface. rust crate extensions and rhai script extensions both receive a view of `ExtensionApi`. the rhai view is a subset plus host functions (see v2.10).

```rust
// crates/inout-ext-builtin-tools/src/lib.rs
use inout_core::{Extension, ExtensionApi};

#[derive(Default)]
pub struct BuiltinToolsExtension;

impl Extension for BuiltinToolsExtension {
    fn name(&self) -> &'static str { "builtin-tools" }
    fn version(&self) -> &'static str { "0.3.0" }
    fn register(&self, api: &mut ExtensionApi) {
        api.tools.register(ReadTool);
        api.tools.register(WriteTool);
        api.tools.register(EditTool);
        api.tools.register(GrepTool);
        api.tools.register(BashTool);
    }
}
```

### rhai script extension loader

file: `crates/inout-core/src/scripting.rs`

```rust
pub fn load_script_extensions(api: &mut ExtensionApi, paths: &[PathBuf]) {
    for dir in paths {
        let dir = expand_tilde(dir);
        if !dir.is_dir() { continue; }
        for entry in std::fs::read_dir(dir).unwrap_or_default() {
            let path = entry.path();
            if path.extension().and_then(|s| s.to_str()) == Some("rhai") {
                if let Ok(ext) = ScriptExtension::from_file(&path) {
                    ext.register(api);
                }
            }
        }
    }
}
```

scripts are parsed at startup. a parse error logs and skips the script. `ScriptExtension` wraps the parsed ast and an isolated `Engine`. each script gets its own `Engine` so one crashing script does not poison another.

### script extension wrapper

file: `crates/inout-core/src/scripting.rs`

```rust
pub struct ScriptExtension {
    name: String,
    engine: rhai::Engine,
    ast: rhai::AST,
}

impl ScriptExtension {
    pub fn from_file(path: &Path) -> Result<Self>;
}

impl Extension for ScriptExtension {
    fn name(&self) -> &'static str;
    fn version(&self) -> &'static str;
    fn register(&self, api: &mut ExtensionApi);
}
```

`ScriptExtension::register` constructs the rhai `api` object (a map) and calls the script's `register(api)` function. if the script has no `register` function, it logs a warning and is skipped.

## registration order and override semantics

extensions register in the order they are loaded by the binary crate. rust extensions load first; rhai scripts load after. this matters for hooks that chain (transform events) or block (early-exit events):

1. first-party rust extensions (builtin-tools, jail, permissions, sessions, context, skills, secret-scanner, observability, undo, http-provider).
2. feature-gated rust extensions (ast, mcp) if enabled.
3. user rhai scripts from `~/.inout/extensions/*.rhai`.
4. project-local rhai scripts from `.inout/extensions/*.rhai`.

last-loaded extension wins on transform events. a user rhai script can override a first-party hook by returning the last value. a user script can also block earlier by returning `block: true` on `tool_call`.

to remove a first-party tool, a user script can call `api.unregister_tool("read")`. the `ExtensionApi` exposed to scripts includes `unregister_tool` and `unregister_command`.

## extension naming

- rust crate: `inout-ext-<kebab-name>`.
- extension name returned by `Extension::name()`: `<kebab-name>`.
- rhai script file name: `<kebab-name>.rhai`. the script's registered name is the file stem.

## acceptance

- [ ] `HookBus::emit` fires all handlers in order; `observe` handlers see events but cannot mutate.
- [ ] transform events chain: `context` handler B sees A's output.
- [ ] block events early-exit: `tool_call` returns on first `block: true`.
- [ ] `ToolRegistry::set_active` rejects unknown names and duplicates.
- [ ] `Tool` trait default methods compile without overrides.
- [ ] rust `Extension::register` can add a tool, a command, and a hook in one call.
- [ ] rhai `ScriptExtension` parses a `.rhai` file and calls its `register(api)` function.
- [ ] rhai script can register a tool and dispatch it via `ToolRegistry`.
- [ ] rhai script can register a `tool_call` hook that blocks a tool.
- [ ] rhai script can unregister a first-party tool.
- [ ] core crate builds with no extension crates on its dependency list.
- [ ] `cargo clippy --all-targets -- -D warnings` green.
