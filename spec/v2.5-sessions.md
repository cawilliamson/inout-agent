## scope

session is all durable agent state, not just transcript. append-only entry tree. branching. compaction. continuity handoff between sessions.

### in scope

1. `SessionEntry` tree — message, model_change, thinking_level_change, active_tools_change, compaction, branch_summary, custom, custom_message, label, session_info, leaf.
2. `SessionRepo` trait — append, fork, list, navigate tree.
3. `JsonlSessionRepo` — newline-delimited json, one entry per line.
4. `SqliteSessionRepo` — optional, behind `feature = "sqlite-sessions"`.
5. branching — fork a conversation like a git branch.
6. compaction — in-place summarisation, `SessionBeforeCompact`/`SessionCompact` events.
7. continuity handoff — auto-write `.inout/context.md` at session end, auto-inject at session start.

### not in scope

- multi-device sync (out of scope).
- session sharing/publishing (out of scope).

## design rationale (mined from pi + zap)

### from pi: the session tree

pi `packages/agent/docs/durable-harness.md` and `packages/agent/src/harness/session/` define the model. key insight:

> treat session as all durable agent state, not just transcript history.

pi `SessionTreeEntry` is a discriminated union:
- `MessageEntry`
- `ThinkingLevelChangeEntry`
- `ModelChangeEntry`
- `ActiveToolsChangeEntry`
- `CompactionEntry`
- `BranchSummaryEntry`
- `CustomEntry` / `CustomMessageEntry`
- `LabelEntry`
- `SessionInfoEntry`
- `LeafEntry`

each entry has `id`, `parentId`, `timestamp`. append-only tree, not flat log. branching is natural — fork creates a new leaf with the parent as ancestor. compaction is an entry, not a destructive operation.

pi durability principle: **every accepted mutation must be durable before the public API resolves.** pending writes queue and flush at turn end.

pi recovery model: crash before enqueue = not accepted; crash after enqueue = restored; crash during provider request = mark interrupted, no auto-retry; crash during tool call = append interrupted/error result, retry only if `retry_safe()`.

### from zap: continuity handoff

zap session continuity is the best of any agent. four files maintained automatically:

| file | updated when | contains | loaded when |
|---|---|---|---|
| `INOUT.md` (zap `ZAP.md`) | `/init` (manual, once) | project overview, build commands, architecture, do-not-touch list | every session (on-demand) |
| `.inout/understanding.md` | `/init` + `/understand` | navigation map + business-domain map | every session (on-demand) |
| `.inout/context.md` | end of every session (auto) | last session: goal, files touched, what next | session start (on-demand) |
| `.inout/session_log.md` | every session (auto) | history of all past sessions, indexed by date | on request (on-demand) |

critical: these files are **never pre-loaded into context**. the model reads them only when relevant, using `read_file`. a session about fixing a typo does not pay the cost of loading the entire project architecture.

at session end (zap `src/session/`): summarise last 10 messages via small LLM call -> 1-3 bullet points with concrete next steps (file names, function names) -> write `.inout/context.md` (goal, files touched, what next) -> append `.inout/session_log.md` (dated history).

at session start: "what was being worked on" line appears in startup banner. full `context.md` injected into system prompt as `## Last Session Handoff`.

what we keep: both. the entry tree from pi, the continuity files from zap.

### session repo trait

file: `crates/inout-ext-sessions/src/repo.rs`

```rust
#[async_trait]
pub trait SessionRepo: Send + Sync {
    async fn append_entry(&self, entry: SessionEntry) -> Result<()>;
    async fn fork(&self, target_id: &str, opts: ForkOptions) -> Result<String>;
    async fn list(&self, opts: ListOptions) -> Result<Vec<SessionInfo>>;
    async fn navigate_tree(&self, from: &str, to: &str) -> Result<NavigateTreeResult>;
    async fn build_context(&self, leaf_id: &str) -> Result<SessionContext>;
    async fn set_leaf_id(&self, target_id: &str) -> Result<()>;
    async fn get_metadata(&self) -> Result<SessionMetadata>;
}
```

### jsonl repo

file: `crates/inout-ext-sessions/src/jsonl_repo.rs`

one entry per line, append-only. `build_context` walks from leaf to root, applying `ActiveToolsChange`, `ModelChange`, `Compaction` entries, collecting `Message` entries. fork creates a new jsonl file (or a branch marker in a single file) with `parent_id` pointing to the fork point.

### continuity handoff

file: `crates/inout-ext-sessions/src/continuity.rs`

```rust
pub struct ContinuityFiles {
    pub inout_md: PathBuf,
    pub understanding: PathBuf,
    pub context: PathBuf,
    pub session_log: PathBuf,
}

impl ContinuityFiles {
    pub async fn write_handoff(&self, session: &Session) -> Result<()>;
    pub async fn load_handoff(&self) -> Option<HandoffContext>;
    pub async fn write_session_log_entry(&self, goal: &str, files: &[String]) -> Result<()>;
}
```

`write_handoff` calls the LLM over the last 10 messages for 1-3 next-step bullets. `load_handoff` returns the context to inject into the system prompt as `## Last Session Handoff`.

### compaction

file: `crates/inout-ext-sessions/src/compaction.rs`

```rust
pub struct CompactionSettings {
    pub max_messages: usize,
    pub keep_recent: usize,
    pub summarize_older: bool,
}

impl Session {
    pub async fn compact(&mut self) -> Result<()>;
}
```

emits `SessionBeforeCompact` (cancellable) and `SessionCompact` events via the hook bus. replaces older messages with a summary entry; keeps recent N messages verbatim.

### branching

`/branch <name>` — fork current conversation. `/branches` — list. `/switch <name>` — move between branches. implemented via `SessionRepo::fork` + `navigate_tree`.

## acceptance

- [ ] `inout-core` builds and passes clippy without `inout-ext-sessions`.
- [ ] `SessionEntry` serialises/deserialises round-trip via jsonl.
- [ ] `JsonlSessionRepo::append_entry` is atomic (flush before return).
- [ ] `fork` creates a new leaf with correct `parent_id`.
- [ ] `build_context` walks the tree applying non-message entries correctly.
- [ ] `ContinuityFiles::write_handoff` produces 1-3 next-step bullets via LLM; writes `.inout/context.md`.
- [ ] `load_handoff` at session start injects into system prompt.
- [ ] `compact` emits before/after events; older messages replaced by summary entry.
- [ ] crash before `append_entry` returns = entry not accepted; crash after = entry restored on reload.
- [ ] `cargo clippy --all-targets -- -D warnings` green.
