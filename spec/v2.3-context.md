## scope

how the system prompt is assembled per turn, how the message window slides, how dropped turns are summarised, how the edit ledger survives window eviction, and how casual turns skip overhead.

### in scope

1. `build_system_prompt` — modular assembly with skill injection.
2. `build_casual_system_prompt` — minimal prompt for greetings.
3. `windowed_history` — sliding window (default last 8 user turns).
4. `DropSummarizer` — LLM-summarise turns that slide off the window.
5. `EditLedger` — per-file record of edits, injected into system prompt.
6. casual turn detection.
7. context-fill percentage + compaction trigger at 90%.

### not in scope

- compaction entry persistence (v2.5).
- prompt caching wire format (v2.9).

## design rationale (mined from zap)

zap's `src/context_manager.rs` and `src/session/turn.rs` are the model. key ideas:

1. **casual turn detection.** `is_casual_message` + `needs_prior_context` (from `zap/src/session/casual.rs`). greetings after a model question -> ~31-token prompt, no tools. "yes"/"ok"/"go ahead" after a model question -> full context (it is an answer). pure social -> always casual. saves ~2-4k tokens per greeting.

2. **edit ledger.** `Session::edited_files: HashMap<String, EditedFile>` records every file written or edited. injected into the system prompt on every non-casual turn. the LLM remembers what files it modified even after the original turns slide out of the window. sorted by `last_turn` desc, `ops_count` desc, take top 20. structural fix for a real failure mode.

3. **sliding window + drop summary.** `windowed_history` caps to last 8 user turns. `maybe_summarize_dropped_turns` runs before the inner tool-loop: detects turns that slid off since last turn, calls the LLM with a focused summarisation prompt, appends to `dropped_summary`. the summary is prepended to the windowed history on the next LLM call as a fake user/assistant pair:
   ```
   user: [Context from earlier in this session — turns that slid off the context window]
   <summary>
   assistant: Understood — I have the earlier context.
   ```

4. **summarizer fallback.** `SUMMARIZER_TOOL_RESULT_CAP = 500` chars per tool result (prevents huge reads blowing the summarizer window). `SUMMARIZER_INPUT_CHAR_CAP = 40_000` — if exceeded, fall back to text-only summary (`text_drop_summary` extracts first 200 chars of each user turn + 250 of paired assistant reply + tool names). a slow/unavailable model never blocks the turn.

5. **projected skill token pre-check.** zap computes `projected_skill_tokens` BEFORE the compaction check, so a context that looks fine at 75% but has matched skills pushing it past 100% triggers compaction proactively. prevents LLM context overflow errors.

6. **context-fill percentage.** `ctx_pct` computed against windowed history + dropped_summary + projected skills. at >=90% -> auto-compact. at >=100% with budget set -> refuse turn with notice.

7. **prompt caching (anthropic).** system prompt + tool definitions marked with `cache_control` breakpoints. final turn last message also gets a breakpoint for reuse on next request. (wire format in v2.9.)

what we keep verbatim: all of the above. zap context handling is the tightest of any agent reviewed.

## components

### system prompt assembly

file: `crates/inout-ext-context/src/mod.rs`

```rust
pub fn build_system_prompt(config: &Config, skill_block: &str) -> Result<String>;
pub fn build_casual_system_prompt(config: &Config) -> String;
pub fn build_slm_system_prompt(config: &Config) -> Result<String>;  // optional, for SLM mode
```

modular sections:
- identity (1-2 lines)
- cwd
- core tool rules
- code-nav order (if ast enabled)
- security prose
- sub-agent docs (if spawn enabled)
- reasoning/investigation sections
- INOUT.md project context
- domain-map staleness nudge
- skill block (injected)

`build_casual_system_prompt` omits: code-nav, tool-policy, sub-agent, security, git-status, project context. saves ~6-8k tokens.

### drop summarizer

```rust

    pub async fn maybe_summarize_dropped_turns(&mut self);
```

LLM call with focused summarization prompt for newly-dropped batch. appends to `dropped_summary`. text fallback on failure or input too large.

### edit ledger

file: `crates/inout-ext-context/src/ledger.rs`

```rust
#[derive(Debug, Clone)]
pub struct EditedFile {
    pub path: String,
    pub ops_count: u32,
    pub last_turn: usize,
    pub last_op: String,  // "edit" | "write" | "batch_edit"
    pub last_preview: String,  // first 80 chars of last change
}

impl EditedFile {
    pub fn summary(&self, path: &str) -> String;
    pub fn record_edit(&mut self, turn: usize, op: &str, preview: &str);
}

pub struct EditLedger { files: HashMap<String, EditedFile> }

impl EditLedger {
    pub fn record(&mut self, path: &str, turn: usize, op: &str, preview: &str);
    pub fn render_block(&self) -> Option<String>;  // top 20, sorted by last_turn desc then ops_count desc
    pub fn clear(&mut self);
}
```

injected into system prompt as:
```
-- Edit Ledger --
Files modified this session (persists across context windows):
  <summary lines>
```

### casual detection

file: `crates/inout-ext-context/src/casual.rs`

```rust
pub fn is_casual_message(input: &str) -> bool;
pub fn is_topic_shift(input: &str, messages: &[Message]) -> bool;
pub fn needs_prior_context(input: &str, messages: &[Message]) -> bool;
```

casual = pure greeting/social (`hi`, `hello`, `hey`, `thanks`, `good morning`, `how are you`).
`needs_prior_context` returns true if the last assistant message asked a question AND the input is a short answer (`yes`, `ok`, `go ahead`, `cool`, `sounds good`) — these get full context.
