# inout openspec v2.8 — linting

project: inout
version: 0.3.0-dev
created: 2026-07-19
workspace: all crates

## workspace

lint rules apply to the whole cargo workspace. root `Cargo.toml` declares `[workspace.lints]`. every crate in `crates/*` inherits via `[lints] workspace = true`. `inout-core` must still build and pass clippy with no extension crates on its dependency list.

## scope

the strict rust discipline zap lacks. zap ships no `clippy.toml`, no `rustfmt.toml`, no `[lints]` table in `Cargo.toml`, no `deny.toml`, no pre-commit hook. inout makes the opposite bet: lints are load-bearing, green on every commit.

### in scope

1. `clippy.toml` — strict lint configuration.
2. `rustfmt.toml` — formatting.
3. `[lints]` table in `Cargo.toml` — workspace-level clippy + rust lints as errors.
4. `deny.toml` — `cargo-deny` config for supply chain (advisories, licenses, bans, sources).
5. `.cargo/config.toml` — default flags.
6. pre-commit hook via `cargo-husky` (dev-dep).
7. CI workflow that runs `cargo clippy --all-targets -- -D warnings`, `cargo fmt --check`, `cargo deny check`, `cargo test`.

### not in scope

- miri (out of scope for v2; add for unsafe-heavy modules later).
- fuzzing (out of scope v2).

## design rationale (mined from zap — what it lacks)

zap `Cargo.toml` has zero lint config. zap ships no `clippy.toml`/`rustfmt.toml`. zap contributing guide says "run cargo check and cargo clippy before submitting — zero warnings expected" but enforces nothing. this is the gap inout fixes.

the rust ecosystem has the tooling; zap just did not wire it. we wire it.

```toml
[workspace.lints.rust]

[workspace.lints.clippy]
all = { level = "deny", priority = -1 }
correctness = { level = "deny", priority = -1 }
suspicious = { level = "deny", priority = -1 }
style = { level = "deny", priority = -1 }
complexity = { level = "deny", priority = -1 }
perf = { level = "deny", priority = -1 }
pedantic = { level = "warn", priority = -1 }
nursery = { level = "warn", priority = -1 }
cargo = { level = "warn", priority = -1 }

# specific denies worth calling out
enum_glob_use = "deny"
expect_used = "deny"
unwrap_used = "deny"
dbg_macro = "deny"
print_stdout = "deny"
print_stderr = "deny"
todo = "warn"
unimplemented = "deny"
wildcard_imports = "deny"
module_name_repetitions = "allow"  # common in rust, pedantic otherwise
must_use_candidate = "warn"
```

rationale for `unwrap_used = "deny"` and `expect_used = "deny"`: rust-native means no panics from lazy unwraps. use `?` or `anyhow::Context` everywhere. `cfg(test)` blocks can `unwrap` freely — add `#[allow]` on test modules.

### pre-commit hook

dev-dependency: `cargo-husky = "1"`.

```toml
[dev-dependencies]
cargo-husky = { version = "1", features = ["precommit-hooks", "run-cargo-fmt", "run-cargo-clippy"] }
```

this installs a `.git/hooks/pre-commit` that runs `cargo fmt --check` + `cargo clippy --all-targets -- -D warnings` on every commit. failures block the commit.

### CI workflow

file: `.github/workflows/ci.yml`

```yaml
name: ci
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: rustfmt, clippy
      - run: cargo fmt --check
      - run: cargo clippy --all-targets -- -D warnings
      - run: cargo deny check
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo test --all-features
      - run: cargo test --no-default-features  # v1.0 lean core still builds
```

### allowlist for test code

```rust
#[cfg(test)]
#[allow(clippy::unwrap_used, clippy::expect_used, clippy::dbg_macro)]
mod tests { ... }
```

tests get to unwrap freely. prod code does not.
